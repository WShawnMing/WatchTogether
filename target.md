# WatchTogether 本地同片源同步版需求文档

## 1. 文档目的

这份文档用于定义 `WatchTogether` 项目在 `main` 分支上的正确目标、范围、实现原则、关键流程、边界情况和验收标准。

这份文档的主要读者是后续接手实现的工程代理，例如 Cursor。

文档目标不是描述当前代码“已经做了什么”，而是明确说明：

- 这个项目最终应该做成什么样
- 哪些功能是必须稳定支持的
- 哪些旧逻辑应该删除或重构
- 哪些边缘情况必须提前考虑

## 2. 产品定位

`WatchTogether` 是一个桌面端双人/多人一起看片工具，目标场景是：

- 用户位于同一个局域网或虚拟局域网中
- 双方电脑上都已经拥有同一份本地视频文件
- 应用只负责发现房间、验证片源一致性、同步播放进度
- 应用不负责把视频文件从一端传给另一端
- 字幕文件各自本地使用，不同步、不上传、不校验一致性

核心价值：

- 避免大视频在差网络环境下传输缓慢
- 保留双方各自本地的画质
- 保证暂停、播放、拖动进度时尽量同步

## 3. 当前主方向

`main` 分支必须坚持下面这个方向：

### 3.1 必须保留的模式

- 双方各自拥有相同本地视频文件
- 房主选择本地视频文件
- 其他成员选择自己的本地视频文件
- 系统校验两边文件是否一致
- 校验通过后，各自播放自己的本地文件
- 房间内只同步控制状态和时间轴

### 3.2 必须删除或隔离的旧模式

以下模式不应该继续混在 `main` 的主流程里：

- 房主上传视频给其他人拉流
- 基于房主媒体流分发的缓冲门控
- “已缓存几秒”“准备阈值”“缓冲速率观测”这一类为流式传输设计的 UI 和状态机
- 任何会让用户误以为当前模式还在传输视频的交互

如果未来还要保留旧模式，应单独做成另一个分支或明确的“实验模式”，不能污染 `main` 的主流程。

## 4. 平台与运行环境

### 4.1 平台要求

- 支持 `macOS`
- 支持 `Windows`

### 4.2 网络前提

- 双方已加入同一个局域网或虚拟局域网
- 可使用贝锐蒲公英、ZeroTier、Tailscale 等工具组网
- 项目不依赖公网中心媒体服务器
- 项目允许存在本机房间发现与控制同步服务

### 4.3 Git 要求

- 使用 Git 管理
- 主分支为 `main`
- 旧的转发/拉流方案可保留在单独分支，不应继续主导 `main`

## 5. 核心用户故事

### 5.1 房主建房

1. 用户打开应用
2. 输入昵称
3. 点击“开始共享”
4. 可选设置房间密码
5. 房间会在局域网或虚拟局域网中被其他用户发现

### 5.2 观影方加入

1. 用户打开应用
2. 输入昵称
3. 在“附近房间”中看到房间
4. 如果房间有密码，则输入密码
5. 点击加入
6. 成功进入房间

### 5.3 片源匹配

1. 房主选择本地视频文件
2. 应用计算片源指纹
3. 观影方选择本地视频文件
4. 应用校验双方文件是否一致
5. 一致则进入可播放状态
6. 不一致则明确提示并要求重新选择

### 5.4 一起观看

1. 任一方点击播放
2. 所有人在相近时间开始播放
3. 任一方点击暂停
4. 所有人同步暂停
5. 任一方拖动进度条
6. 所有人同步跳转到目标时间附近

### 5.5 字幕使用

1. 每个人都可以在自己设备上单独选择字幕文件
2. 字幕只在当前设备显示
3. 字幕不会上传
4. 字幕不会同步给别人
5. 字幕不参与房间一致性校验

## 6. 非目标

以下不是当前版本的目标：

- 将视频文件从房主传输给其他成员
- 云端存储视频或字幕
- NAT 打洞、TURN 中继、WebRTC 媒体传输
- 语音聊天
- 截至当前版本的手机端支持

## 7. 功能需求

## 7.1 房间发现与连接

### 7.1.1 自动发现

- 支持自动发现同网络中的房间
- 应兼容常见局域网和虚拟局域网网卡
- 不应只偏向蒲公英某个网段
- 对 UDP 广播不通的 VPN，应该有主动探测兜底

### 7.1.2 手动刷新

- 提供“刷新附近房间”按钮
- 同时保留定时刷新
- 刷新不应明显卡住 UI

### 7.1.3 房间连接稳定性

- 加入房间速度应可接受
- 不应频繁因为网络轻微抖动自动退房
- 重连策略应明确
- 房主离开时，其他人应直接退出房间并给出清晰提示

### 7.1.4 进入/离开提醒

- 成员加入房间时给出低侵入 toast
- 成员离开房间时给出低侵入 toast
- 不能重复弹两次
- 不能在初始快照或重连恢复时误报

## 7.2 本地视频文件匹配

### 7.2.1 文件选择

- 房主和观影方都需要选择本地视频文件
- 支持常见格式：`mp4`、`mov`、`mkv`、`webm`
- 允许扩展更多格式，但前提是播放链路稳定

### 7.2.2 文件一致性校验

至少校验：

- `SHA-256`
- 文件大小
- 视频时长

校验要求：

- 一致时显示“已匹配”
- 不一致时显示“文件不一致”
- 不一致时禁止用户误以为已经能一起看

### 7.2.3 切换片源

切换片源时必须：

- 销毁旧播放器实例
- 清理旧字幕层
- 清理旧播放状态缓存
- 清理旧待应用的远端播放状态
- 重新挂载新片源

切换片源时绝不能出现：

- 白屏卡死
- 旧字幕残留
- 旧进度残留到新片源
- 旧同步状态覆盖新片源

## 7.3 播放器能力

### 7.3.1 最低必须支持

- 正常显示画面
- 正常播放声音
- 暂停/播放可用
- 拖动进度条可用
- 全屏可用
- 键盘快捷键可用

### 7.3.2 音视频兼容性

这是当前最关键的问题，必须优先解决。

要求：

- 不能只依赖 Chromium 原生 `<video>` 对音视频编码的默认支持
- 对 `mkv/hevc/flac` 这类高概率失败的组合必须有稳定兜底
- 不能出现“有画面没声音”
- 不能出现“黑屏但时间轴在动”

### 7.3.3 推荐实现策略

Cursor 实现时应优先考虑下列策略之一：

1. 原生播放器内核方案
   - 例如 `mpv/libmpv`
   - 这是最稳妥的长期方向
   - 更适合解决 `HEVC`、`FLAC`、高规格 `MKV`

2. 本机兼容代理方案
   - 当检测到本地文件音视频编码不兼容时
   - 自动生成本地兼容版本，例如 `H.264/AAC MP4`
   - 播放器播放兼容版本

3. 如果继续使用 HTML5 video
   - 必须通过本机 HTTP Range 服务提供媒体
   - 不要直接依赖奇怪的本地文件路径或不稳定的协议桥接
   - 不能把“能打开视频”误认为“能稳定播放音频和画面”

结论：

- 当前项目若以“稳定可播”为第一优先级，应优先考虑原生播放器方案
- 如果暂时不切原生播放器，至少要让“本机 HTTP Range + 本地兼容转码”稳定落地

## 7.4 字幕需求

### 7.4.1 核心原则

- 字幕是本地私有资源
- 不上传
- 不同步
- 不校验

### 7.4.2 支持格式

- `srt`
- `vtt`
- `ass`
- `ssa`

### 7.4.3 字幕行为

- 每个用户自己选择字幕文件
- 每个用户自己决定是否使用字幕
- 每个用户自己调整字幕样式
- 其他成员的字幕文件和字幕状态不会影响当前用户

### 7.4.4 字幕设置

至少支持：

- 字号
- 上下位置

位置要求：

- 字幕设置入口应集成在播放器内部
- 不应作为页面上方的大块设置卡片长期占位

### 7.4.5 容错要求

字幕绝不能导致整应用白屏或崩溃。

必须覆盖：

- 扩展名与内容不匹配
- 非文本文件误选成字幕
- 解析失败
- ASS 渲染失败
- 切换视频后旧字幕实例未清理

处理要求：

- 字幕失败时只影响字幕，不影响视频播放
- 给出可理解错误提示
- 保留重新选择字幕的能力

## 7.5 播放同步

### 7.5.1 需要同步的动作

- 播放
- 暂停
- 拖动进度条
- 重播
- 可选：播放速率

### 7.5.2 同步原则

- 房间成员各自播放自己的本地视频文件
- 只同步时间轴和播放控制
- 同步优先级应低于本地用户当前正在拖动的行为
- 不能在用户拖动时立刻被旧状态拉回去

### 7.5.3 拖动要求

- 本地拖动时，短时间内应屏蔽旧远端状态回写
- 拖动结束后，再以较新状态同步
- 快照或心跳不能覆盖比当前更旧的播放状态

### 7.5.4 房间内播放状态更新策略

- 使用明确的播放状态版本或时间戳
- 只接受“更新更晚”的状态
- 避免 `room snapshot` 和 `playback state` 相互打架

## 7.6 UI/UX 要求

### 7.6.1 总体风格

- 简约
- 克制
- 接近 Apple / Google 风格
- 去掉无意义的技术信息

### 7.6.2 页面层级

页面应围绕三件事组织：

- 房间发现与加入
- 当前房间状态
- 播放器

### 7.6.3 不应出现的 UI

- 流式传输模式遗留的“缓存观测”“准备阈值”“缓存几秒”
- 用户无感知价值的内部状态
- 页面上方长期占位的大块字幕设置卡片

### 7.6.4 必须保留的状态提示

- 当前是否已加入房间
- 当前片源是否已匹配
- 文件是否不一致
- 房主是否已离开
- 当前字幕是否已在本地载入

### 7.6.5 播放器交互

- 全屏时控制栏行为应接近成熟播放器
- 鼠标移动时显示控制栏
- 静止时自动隐藏
- 键盘快捷键支持常见习惯

## 8. 架构要求

## 8.1 保留的服务职责

本机 relay 服务仍然有价值，但职责应该收敛为：

- 房间发现
- 房间加入/退出
- 房间成员管理
- 播放状态同步
- 本机媒体兼容代理或本机 HTTP Range 服务

## 8.2 不该继续承担的职责

- 房主向其他成员分发视频文件
- 基于远端拉流的缓冲门控
- 媒体传输速度观测驱动的播放状态机

## 8.3 代码组织建议

建议重构为清晰分层：

1. `Room / Discovery Layer`
   - 发现、加入、断线、重连、成员状态

2. `Playback Sync Layer`
   - 统一处理播放状态、时间戳、冲突解决

3. `Local Media Layer`
   - 本地文件选择
   - 文件指纹计算
   - 媒体兼容性探测
   - 本机稳定播放入口

4. `Subtitle Layer`
   - 本地字幕载入
   - ASS/VTT 渲染
   - 样式控制

当前项目最需要避免的是：

- UI 层直接承担太多媒体状态机逻辑
- 旧模式和新模式混在一起
- 同一个状态同时由多个来源更新

## 9. 关键边缘情况

Cursor 必须明确覆盖以下边缘情况：

### 9.1 文件与播放

- 选择文件后没有声音
- 选择文件后黑屏
- 切换片源时白屏
- 切换片源时旧播放器没销毁
- 房间播放到片尾后再次点击播放无法重播

### 9.2 同步

- 拖动时被拉回原位置
- 本地正在拖动时被旧状态覆盖
- 重连后旧状态覆盖新状态
- 房主离开后观影方卡在 websocket 错误状态

### 9.3 字幕

- 字幕文件格式错误
- 字幕内容损坏
- 字幕渲染失败
- 视频切换后字幕层残留
- 本地字幕影响别人

### 9.4 房间状态

- 成员加入提醒重复弹
- 成员离开不提醒
- 房间发现过慢
- 发现列表中状态过旧

## 10. 验收标准

## 10.1 最低可交付标准

以下 3 点必须先稳定成立：

1. 能稳定播放本地视频
2. 能稳定同步进度
3. 能各自使用自己的本地字幕

如果这 3 点有任何一项不稳，其他附加功能一律不算完成。

## 10.2 功能验收

### 用例 A：双机同片源

- 两台机器进入同一房间
- 房主选择本地视频
- 观影方选择相同本地视频
- 校验通过
- 双方都能正常播放

通过标准：

- 有画面
- 有声音
- 不黑屏

### 用例 B：暂停/播放同步

- 一方播放
- 另一方同步播放
- 一方暂停
- 另一方同步暂停

通过标准：

- 不应明显偏差

### 用例 C：拖动同步

- 一方拖动到新时间点
- 另一方跳转到相近位置

通过标准：

- 不出现明显回跳
- 不出现刚拖动完又被拉回旧位置

### 用例 D：切换片源

- 在房间中重新选择另一个本地视频

通过标准：

- 播放器不白屏死机
- 旧字幕不残留
- 新片源可正常播放

### 用例 E：本地字幕

- 双方分别选择不同字幕文件

通过标准：

- 各自显示自己的字幕
- 不互相影响
- 字幕失败不影响视频

### 用例 F：房主离开

- 房主关闭房间或退出

通过标准：

- 其他成员明确收到房间已关闭提示
- 自动退出房间
- 不停留在错误重连状态

## 11. 实现优先级

建议 Cursor 按以下顺序推进：

### P0

- 稳定播放本地视频
- 解决无声
- 解决黑屏
- 解决切换片源白屏
- 解决拖动进度回跳

### P1

- 清理旧流式状态机
- 稳定房间发现
- 稳定成员加入/退出提醒
- 本地字幕稳定加载与调整

### P2

- UI 精修
- 更多播放器快捷键
- 更细的错误提示

## 12. 对 Cursor 的明确执行要求

Cursor 在实现时必须遵守：

1. 不要继续在 `main` 中混入“房主传视频给别人”的流式思路。
2. 先保证“播放视频、有声音、同步进度、本地字幕”这 4 件事稳定。
3. 每次改动后都要回归以下路径：
   - 选片
   - 播放
   - 暂停
   - 拖动
   - 切换片源
   - 加载本地字幕
4. 不要只看用户指出的一小段代码，要从“播放链路、同步链路、字幕链路、房间链路”整体排查。
5. 对旧逻辑要敢删，不能为了“兼容”继续保留已经不适合当前模式的状态机。

## 13. 最终一句话目标

这个项目当前版本的唯一核心目标是：

**在同一局域网或虚拟局域网中，让双方各自播放自己本地的同一份视频文件，并稳定同步播放进度，同时各自独立使用自己的本地字幕。**
